# admin-auth

The login endpoint and JWT machinery for the single administrator. The
credential itself (env-supplied, optional password with auto-gen) is
covered by **`auth-bootstrap`** â€” this spec assumes that credential is
already loaded into `cfg.Admin` and focuses on the login surface, token
audience, and isolation from the user portal.

## Purpose & boundaries

The dashboard recognizes exactly one administrator, identified by
`ADMIN_USERNAME`, with the password sourced from `ADMIN_PASSWORD` (or
auto-generated by the bootstrap path â€” see `auth-bootstrap`). This
module covers:

- The admin login HTTP endpoint.
- The admin JWT shape (audience, expiry).
- Strict isolation from user-domain auth (separate endpoints, separate
  audiences, no token cross-acceptance).

## Requirements

### Requirement: Single Administrator With No Database Row

The system SHALL recognize exactly one administrator, sourced from
environment configuration via the `auth-bootstrap` module. There SHALL
be no `admins` table, no admin registration flow, and no admin
moderation endpoint.

#### Scenario: Admin credential loaded from environment

- **WHEN** the backend starts with `ADMIN_USERNAME` set
- **THEN** that value defines the only administrator
- **AND** no `admins` table or admin database record is created
- **AND** the password used to authenticate is whatever `auth-bootstrap` exposes via `cfg.Admin.Password` (either the operator-set value or the boot-generated one)

#### Scenario: No admin registration endpoint

- **WHEN** any HTTP request attempts to create, register, list, or modify admin accounts
- **THEN** the system SHALL reject it â€” no such endpoint exists

### Requirement: Admin Console Login

The system SHALL provide a dedicated admin login endpoint, separate
from the user portal, that authenticates against the environment
credential.

#### Scenario: Valid admin login

- **WHEN** a client POSTs to `/api/admin/auth/login` with `{username, password}` matching the configured `ADMIN_USERNAME` and the active `ADMIN_PASSWORD`
- **THEN** the system SHALL respond HTTP 200 with a signed JWT
- **AND** the JWT claims SHALL include audience `admin` and a finite expiry (`Auth.AccessTokenTTL`, default 24h)

#### Scenario: Invalid admin login

- **WHEN** the submitted credentials do not match
- **THEN** the system SHALL respond HTTP 401 with a generic error message that does NOT reveal whether the username was correct

#### Scenario: Constant-time password comparison

- **WHEN** the submitted admin password is compared against the configured value
- **THEN** the comparison SHALL run in constant time to avoid timing-side-channel disclosure

#### Scenario: Admin login isolated from OIDC

- **WHEN** the admin login endpoint is invoked
- **THEN** it SHALL NOT invoke any OIDC or email/password user-auth path
- **AND** admin authentication SHALL depend only on the environment credential

### Requirement: Separated Admin and User Auth Domains

Admin and user authentication SHALL be fully isolated: separate
endpoints, separate JWT audiences, and no cross-acceptance of tokens.

#### Scenario: User token rejected on admin routes

- **WHEN** a token issued by the user portal (audience `user`) is presented to any `/api/admin/*` route
- **THEN** the system SHALL respond HTTP 403 regardless of the token's signature validity

#### Scenario: Admin token rejected on user routes

- **WHEN** an admin token (audience `admin`) is presented to a `/api/user/*` route
- **THEN** the system SHALL respond HTTP 403

#### Scenario: Independent sessions in the same browser

- **WHEN** a user is signed in as admin and ALSO signs in as a portal user via the unified `/login`
- **THEN** the SPA SHALL store both tokens in separate storage keys (`adminAuthStore` and `portalAuthStore`)
- **AND** neither session SHALL invalidate the other

### Requirement: Admin Session Lifetime

Admin JWTs SHALL carry an expiry and be re-issued only through admin login.

#### Scenario: Expired admin token

- **WHEN** an admin presents an expired admin JWT to a protected route
- **THEN** the system SHALL respond HTTP 401
- **AND** the SPA SHALL redirect to `/login` with `?next=` preserving the protected path

#### Scenario: Admin credential rotation

- **WHEN** the `ADMIN_PASSWORD` environment value is changed (or auto-rotated by a restart without `.env` persistence â€” see `auth-bootstrap`)
- **THEN** previously issued admin tokens SHALL remain valid only until their own natural expiry
- **AND** new logins SHALL require the new password
- **AND** the JWT signature verification path SHALL be unaffected (signatures use `JWT_SECRET`, not the password)
